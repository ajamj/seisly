# Plan v1.1-03: Worker Resource Hardening - Summary

## Objective

Harden the Python worker against hangs and resource exhaustion.

**Purpose:** Python plugins may run into infinite loops or consume too much RAM. The host must remain stable and be able to recover.

**Output:** Background watchdog and OS-level process quotas.

**Implementation Details:**

1. **Background Heartbeat Thread** (`ipc.rs`):
   - Added `HEARTBEAT_INTERVAL_SECS` constant set to 5 seconds
   - Added `MAX_HEARTBEAT_FAILURES` constant set to 3 consecutive failures
   - Spawned background thread in `IpcBridge::new()` that:
     - Sends `ping` request to worker every 5 seconds
     - Increments failure counter on timeout/error
     - Resets failure counter on successful ping response

     - Logs warning messages at each failure level

2. **Memory Monitoring** (`ipc.rs`):
   - Integrated `sysinfo` crate for cross-platform process monitoring

3. **Worker State Management** (`ipc.rs`):
   - Added `heartbeat_stop` atomic flag for graceful shutdown
   - Added `failure_count` atomic counter for tracking consecutive failures
   - Added `worker_pid` storage for process monitoring
   - Added `system` mutex for thread-safe sysinfo access
   - Implemented `execute_internal_simple()` for heartbeat pings (no ID tracking)

4. **Worker PID Capture** (`ipc.rs`):
   - Modified `ensure_worker()` to capture worker process ID
   - Stores PID in `worker_pid` for monitoring thread access
   - Logs worker PID for debugging

## Technical Notes

### Heartbeat Thread Design
- Runs in separate thread to avoid blocking main UI
- Uses atomic operations for thread-safe state management
- Creates temporary `IpcBridge` instance for ping execution
- Respects `heartbeat_stop` flag for graceful shutdown

### Memory Monitoring Strategy
- Dual-layer protection:
  1. **OS-level**: Hard caps via prlimit (Linux)
  2. **Application-level**: Watchdog monitoring (all platforms)
- 512 MB default limit balances functionality and safety
- Warning logs provide visibility into memory usage patterns

### Windows Job Objects (Deferred)
- Full implementation requires complex Windows API setup
- Current watchdog provides adequate protection for development
- Can be enhanced later with proper Job Object integration
- Foundation in place for future enhancement
