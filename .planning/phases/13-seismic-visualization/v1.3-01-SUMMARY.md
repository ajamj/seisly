## What Was Accomplished

### WU-1301: Extend ColormapPreset with Rainbow and BlueWhiteRed

**State before:** The `Rainbow` and `BlueWhiteRed` variants were already declared in the `ColormapPreset` enum with `generate_preset_data` match arms, but they were **not initialized** in `ColormapManager::new()` ŌĆö the initialization loop only covered Seismic, Viridis, Magma, and Gray. No unit tests existed.

**Changes made:**
- Added `ColormapPreset::Rainbow` and `ColormapPreset::BlueWhiteRed` to the initialization loop in `ColormapManager::new()` (file: `crates/seisly_render/src/colormaps.rs`)
- Added 7 unit tests:
  - `test_rainbow_colormap_has_256_entries` ŌĆö verifies 1024 bytes (256 x RGBA)
  - `test_rainbow_first_entry_is_red` ŌĆö verifies [255, 0, 0, 255]
  - `test_rainbow_last_entry_is_violet` ŌĆö verifies G=0, B=255, A=255
  - `test_bluewhitered_has_256_entries` ŌĆö verifies 1024 bytes
  - `test_bluewhitered_first_entry_is_blue` ŌĆö verifies [0, 0, 255, 255]
  - `test_bluewhitered_last_entry_is_red` ŌĆö verifies [255, 0, 0, 255]
  - `test_bluewhitered_midpoint_is_white` ŌĆö verifies indices 127 and 128 are near-white (>= 252 for RGB)

**Deviation from plan:** The plan specified exact equality assertions for the midpoint test (`assert_eq!(data[512..515], [255, 255, 255])`). Due to floating-point interpolation at the boundary between the two lerp segments (blue-to-white and white-to-red), index 128 produces R=253 instead of 255. The test was adjusted to use a tolerance check (>= 252) for both indices 127 and 128, which correctly validates the white transition region.
