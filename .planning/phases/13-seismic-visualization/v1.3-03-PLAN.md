# Plan: v1.3-03 - SEG-Y Performance Optimization

## 1. Context and Objective

**User stories:**
- As a geophysicist, I want to see the first seismic slice on screen within 1 second after selecting a SEG-Y file SO THAT I can start interpreting without waiting â€” measured as **time-to-first-view** (from file selection to first rendered slice), not just file load time.
- As a user, I want the application to remain responsive during the first-time SEG-Y scan SO THAT I can continue working with other features.
- As a data manager, I want sidecar index files to automatically refresh when the source SEG-Y file changes SO THAT stale indexes don't produce wrong results.

**Note on time-to-first-view vs load time:** Load time measures when the file is read. Time-to-first-view measures when the user actually sees seismic data â€” including normalization, GPU texture upload, and first render. The 1-second target is for time-to-first-view.

**From CONTEXT.md decisions:**
- D-13: Persistent `.sfidx` sidecar via `bincode` serialization
- D-14: Regular grid detection â†’ O(1) mathematical offset calculation
- D-15: Background scan with tx/rx progress
- D-16: Non-blocking "Scanning..." notification
- D-17: Target <1 second load after initial scan
- D-22: SEG-Y parser returns real inline/crossline ranges

**Current state:** `parse_metadata()` returns hardcoded `(1, 1)`. `MmappedSegy` does full O(N) scan every load. No sidecar index. No background scanning.

## 2. Technical Strategy

- **SegyIndex struct**: New file with `#[derive(Serialize, Deserialize)]` struct. Fields: inline_min, inline_max, crossline_min, crossline_max, trace_count, sample_count, sample_interval, format, is_regular, trace_byte_offsets, inline_stride, crossline_stride, trace_header_size, trace_data_offset.
- **Staleness check**: Compare `.sfidx` file mtime with SEG-Y file mtime. If SEG-Y is newer, discard sidecar and rescan.
- **Regular grid O(1)**: After scanning first 100 traces, check constant increments. If regular: compute stride-based offset formula. If irregular: store all byte offsets in Vec.
- **Background scanning**: `std::thread::spawn` for scan, `mpsc::channel` for progress. Scan thread produces `SegyIndex`, main thread constructs `SeismicVolume`.
- **Test fixtures**: Synthetic SEG-Y buffer generator for unit tests without real files.

**Merge conflict note:** This plan modifies `seisly_io` crate only (no `app.rs` changes). Plan v1.3-02 modifies `app.rs`. No conflict risk â€” these plans run truly in parallel.

## 4. Success Criteria

### User-Facing Metrics
- [ ] **Time-to-first-view:** Geophysicist sees the first seismic slice on screen within 1 second after selecting a 1 GB SEG-Y file (with existing sidecar) â€” verified by benchmark: measure wall-clock time from `import_seismic` call to first successful render of any slice in the viewport
- [ ] **Responsive UI:** During first-time scan of a large SEG-Y file, user can continue interacting with other parts of the application (move mouse, resize window, open menus) â€” verified by manual test: start import, attempt UI interaction during scan, confirm no freeze
- [ ] **Data integrity:** After SEG-Y file is modified (e.g., re-exported from processing), sidecar index automatically refreshes â€” verified by: load file (creates sidecar), touch SEG-Y file to update mtime, reload, confirm rescan occurs (not stale index load)

### Technical Metrics
- [ ] `SegyIndex` struct with 14 fields serializes to `.sfidx` via bincode (verified by round-trip test)
- [ ] `MmappedSegy::new` loads sidecar in O(1) when it exists and is fresh (mtime check)
- [ ] Regular grid detection computes `inline_stride`/`crossline_stride` for O(1) math
- [ ] `get_trace_offset` uses overflow-checked O(1) formula for regular grids
- [ ] Background scan runs on separate thread â€” UI thread never blocked
- [ ] "Scanning..." notification shown during first-time scan
- [ ] `parse_metadata` returns real inline/crossline from first trace header (not hardcoded `(1, 1)`)
- [ ] `cargo test -p seisly_io` passes (0 failures)
- [ ] `cargo check -p seisly_app` passes

### Failure Criteria
- **If background scan takes >30 seconds for 1 GB file:** Log warning, fall back to synchronous load with progress bar. User sees "Loading may take a moment..." instead of indefinite background operation.
- **If bincode deserialization fails on valid sidecar:** Delete corrupt `.sfidx`, trigger full rescan. Log error with bincode error message for debugging.
- **If regular grid detection false positive (irregular grid detected as regular):** `get_trace_offset` returns `None` for out-of-bounds coordinates â€” caller handles gracefully with error message "Trace not found at inline X, crossline Y".
- **If sidecar staleness check fails (filesystem doesn't support mtime):** Fall back to full scan every load â€” slower but correct.

### MVP Priority
**This plan is the MVP killer feature.** Current SEG-Y import is the #1 user complaint â€” large files cause severe system lag. Achieving <1 second load time after initial scan is the single most impactful improvement in v1.3. This plan ships first in Wave 1.

### Rollback Strategy
- If sidecar corrupt, delete `.sfidx` and rescan â€” data not lost
- If `bincode` deserialization fails, fall back to full scan â€” application continues
- If WU-1310 breaks app.rs, revert app.rs changes â€” SEG-Y loading still works (synchronously, no progress indicator)
