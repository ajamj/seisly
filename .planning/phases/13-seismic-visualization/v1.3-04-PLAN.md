# Plan: v1.3-04 - Stability Fixes: Float Safety & SafeMmap Documentation

## 1. Context and Objective

**User stories:**
- As a user working with well log data, I want the application to handle missing or NaN depth values gracefully SO THAT my interpretation session is not interrupted by unexpected crashes when sorting wells or trajectories.
- As a developer maintaining this codebase, I want SafeMmap's unsafe boundary clearly documented SO THAT future maintainers understand the safety invariant and don't introduce buffer overflows that could corrupt user data.

**From CONTEXT.md decisions:**
- D-20: Replace all `.unwrap()` in rendering/data paths
- D-21: CRS transform fails loudly (handled in v1.3-04b — separate plan)
- D-22: SEG-Y parser returns real ranges (handled in v1.3-03)

**Scope of THIS plan:** Float comparison NaN safety + SafeMmap documentation only. CRS transformer changes are in **v1.3-04b** (separate plan, Wave 2, depends on this plan).

**Why Wave 1:** These fixes have zero external dependencies — they can ship immediately without waiting for any other plan.

## 2. Technical Strategy

- **Float comparisons**: Replace `partial_cmp().unwrap()` with `total_cmp()` (stable in Rust 1.62+) or `partial_cmp().unwrap_or(Ordering::Equal)` for broader compatibility. NaN values are sorted to the end, preventing panic while maintaining meaningful sort order.
- **SafeMmap docs**: Add `/// # Safety` documentation explaining the invariant that callers must not access beyond the mapped region. Add `#[allow(unsafe_code)]` annotation with comment explaining why `memmap2::Mmap::map()` requires unsafe.

**Impact Analysis:** All modified files are within `seisly_core`. The `total_cmp()` replacement returns the same `Ordering` type — no public API signature changes. Downstream crates (`seisly_io`, `seisly_app`, `seisly_render`) are unaffected.

      ```rust
      /// Returns the entire mapped region as a byte slice.
      ///
      /// # Safety
      /// This method is unsafe because it exposes the raw memory mapping.
      /// Callers must ensure:
      /// 1. The file is not modified or truncated while the mapping is active
      /// 2. The returned slice is not used after the SafeMmap is dropped
      /// 3. No concurrent writes occur to the mapped region
      ///
      /// Prefer using `get(offset, length)` or `get_slice(range)` instead,
      /// which provide bounded, safe access.
      ```

## 4. Success Criteria

### User-Facing Metrics
- [ ] **Crash prevention:** User can load and sort well/trajectory data containing NaN depth values without application crash — verified by unit test with NaN inputs
- [ ] **Developer clarity:** New contributor can understand SafeMmap's safety invariant by reading documentation alone — verified by cargo doc producing no warnings

### Technical Metrics
- [ ] Zero `partial_cmp().unwrap()` calls in well.rs and trajectory.rs
- [ ] Sorting with NaN depth values completes without panic (verified by unit tests)
- [ ] SafeMmap::map() has `#[allow(unsafe_code)]` with documented safety rationale
- [ ] as_slice() has `# Safety` documentation
- [ ] `cargo test -p seisly_core` passes (0 failures)
- [ ] `cargo doc -p seisly_core --no-deps` passes

### Failure Criteria
- **If `total_cmp()` unavailable (older Rust version):** Fall back to `partial_cmp().unwrap_or(Ordering::Equal)` — same NaN safety, broader compatibility.
- **If documentation changes break cargo doc:** Revert safe_mmap.rs changes — no functional impact, safety invariant remains undocumented but code is still safe.

### MVP Priority
**This plan is MVP-enabling for data integrity.** NaN crashes during well sorting would corrupt user sessions and erode trust in the application. SafeMmap documentation prevents future buffer overflow vulnerabilities. Both are foundational quality improvements.

### Rollback Strategy
- If total_cmp causes compilation error (older Rust version), revert to `partial_cmp().unwrap_or(Ordering::Equal)` — same safety, broader compatibility
- If documentation changes break cargo doc, revert safe_mmap.rs — no functional impact
