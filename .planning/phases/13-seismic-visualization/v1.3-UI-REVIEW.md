# UI Review -- Phase v1.3: Seismic Visualization & Plotting

## Scores

| Pillar | Score | Max | Notes |
|--------|-------|-----|-------|
| Copywriting | 2 | 4 | Inconsistent labels ("Time" vs "Time Slice"), emoji in progress indicator, hardcoded status bar text |
| Visuals | 2 | 4 | No colormap selector in UI despite 6 colormaps implemented, horizontal toolbar crowded, no visual legend |

## Detailed Findings

### Copywriting (2/4)

1. **Inconsistent slice label** (`viewport.rs:66`): The slider text is `"Time"` but the enum variant is `SliceType::TimeSlice` and the tab title is `"TimeSlice"`. Should be `"Time Slice"` (two words) everywhere for consistency with `"Inline"` and `"Crossline"`.

2. **Emoji in progress indicator** (`viewport.rs:118-119`): Uses `"Auto-Tracking:"` with a separate emoji rendered via `ui.label()`. The emoji string (visible in code comment area) mixes iconography with text labels. Better to use egui's built-in spinner widget or a consistent icon system.

3. **Hardcoded status bar** (`app.rs:786-788`): Status bar shows `"Ready"` and `"X: 250.5  Y: 312.8  Z: 1523m "` as hardcoded literals. These are placeholder values that never update -- misleading to users. Should either show real cursor coordinates or be removed until mouse tracking is implemented.

4. **Slider labels lack context** (`viewport.rs:101-107`): `"Gain:"` and `"Clip:"` labels have no units, no displayed values (`show_value(false)`), and no tooltips. Users cannot tell what the current gain/clip value is without guessing from slider position.

5. **Action labels are adequate** (`viewport.rs:60-66`): `"Map View"`, `"Section View"`, `"Inline"`, `"Crossline"` are clear and action-oriented. Good.

### Visuals (2/4)

1. **No colormap selector in UI**: Six colormaps are implemented (`Seismic`, `Viridis`, `Magma`, `Gray`, `Rainbow`, `BlueWhiteRed` in `colormaps.rs:9-15`), but there is **no UI control** to switch between them. The `ViewportWidget.colormap` field defaults to `ColormapPreset::Seismic` and is never exposed to the user. This is the single biggest gap.

2. **Duplicate colormap**: `Seismic` and `BlueWhiteRed` (`colormaps.rs:97-105`) generate identical data -- BlueWhiteRed is literally a copy of Seismic with a comment `"Same as Seismic -- explicit alias"`. This wastes UI mental space if both are eventually exposed.

3. **Horizontal toolbar overcrowding** (`viewport.rs:57-128`): The entire control bar (view mode, slice type, slice slider, gain, clip, wiggles, progress) is a single `ui.horizontal()`. On narrower windows this will overflow or wrap unpredictably. Should be split into logical groups or use a wrapping layout.

4. **No visual colormap legend/swatch**: Even if a selector were added, there is no colorbar/swatch rendered alongside the viewport to show the current amplitude-to-color mapping. Standard in seismic interpretation software (e.g., OpendTect, Petrel).

5. **Viewport border is debug-level** (`viewport.rs:231-232`): A `DARK_GRAY` rectangle stroke is drawn around the viewport "as a visual fallback to confirm the widget area is correctly allocated." This is a development artifact that should be removed or made theme-aware.

6. **Progress indicator uses ProgressBar but has no spinner** (`viewport.rs:118-128`): The progress bar appears but there is no animated spinner or throbber to complement it -- only text percentage.

### Color (3/4)

1. **Rainbow colormap is not colorblind-safe** (`colormaps.rs:128-143`): The standard spectral rainbow (Red-Orange-Yellow-Green-Cyan-Blue-Violet) is known to be problematic for deuteranopia and protanopia users. Viridis and Magma are perceptually uniform and colorblind-safe, but Rainbow is still the default in many geoscience tools and may mislead interpretation.

2. **Theme system is well-designed** (`style.rs:16-105`): Dark and light themes use semantic tokens (VS Code-style naming), with proper contrast ratios. Dark theme: text_primary (#FFFFFF) on editor_bg (#1E1E1E) = 16.9:1 contrast (WCAG AAA). Light theme: text_primary (#333333) on editor_bg (#FFFFFF) = 12.6:1 (WCAG AAA). Good.

3. **Seismic colormap (Blue-White-Red) diverging** (`colormaps.rs:100-105`): Pure blue (#0000FF) to pure red (#FF0000) through white. The blue endpoint has very low luminance and may be hard to distinguish from black backgrounds in dark mode. Consider using a slightly brighter blue (e.g., #0000CC -> #3366FF) for dark mode visibility.

4. **Pick point outlines** (`viewport.rs:306`): Horizon picks get a white circle stroke (`Color32::WHITE`) for visibility. This works on dark backgrounds but may have insufficient contrast on light mode. Should be theme-aware (e.g., black stroke on light theme).

### Typography (2/4)

1. **No monospace for numeric display** (`style.rs:137`): The `status()` typography helper correctly uses `FontFamily::Monospace`, but it is **never called** in the viewport or status bar. The status bar uses `FontId::proportional(10.0)` with hardcoded text (`app.rs:788`). Slider values are hidden entirely (`show_value(false)`).

2. **Inconsistent font sizes** (`viewport.rs:229`): The depth-mode placeholder uses `FontId::proportional(20.0)` directly, bypassing the typography module. Should use `typography::heading()` for consistency.

3. **Sidebar headings use hardcoded styling** (`app.rs:400-408`): Labels like `"EXPLORER"`, `"INTERPRETATION"` use `RichText` with `.strong()` but no consistent size. Should reference `typography::heading()`.

4. **No zoom-level adaptation**: All font sizes are absolute. egui supports scaling via `ctx.set_pixels_per_point()`, but the codebase does not adjust relative sizes for different DPI/zoom settings.

### Spacing (3/4)

1. **Theme enforces 4px item spacing** (`style.rs:159`): `style.spacing.item_spacing = egui::vec2(4.0, 4.0)` provides consistent base spacing. Good.

2. **Panel margins are consistent** (`app.rs:798`): Bottom panel uses `inner_margin(8.0)`, sidebar uses `inner_margin(8.0)` (`app.rs:397`). Consistent 8px padding system.

3. **Spacing module is well-organized** (`style.rs:116-122`): Named constants for `ACTIVITY_BAR_WIDTH`, `SIDEBAR_DEFAULT_WIDTH`, `STATUS_BAR_HEIGHT`, `ITEM_SPACING`, `BUTTON_PADDING`. Good maintainability.

4. **Viewport toolbar has no explicit spacing overrides** (`viewport.rs:57`): Relies on egui defaults within the horizontal layout. Sliders, separators, and checkboxes are placed with default spacing, which may feel tight for the number of controls present.

5. **`add_space()` calls in well logs** (`app.rs:493-497`): Manual `ui.add_space(4.0)` and `ui.add_space(8.0)` calls instead of using `spacing::` constants. Minor inconsistency.

### Experience Design (2/4)

1. **Undo/redo is implemented** (`shortcuts.rs:6-17`): `Cmd+Z` for undo, `Cmd+Y` for redo. Commands are used for picks, auto-tracking, and fault sticks. Good TDD-compliant implementation.

2. **No colormap keyboard shortcut**: No way to cycle colormaps from the keyboard. Given that colormaps are a core visualization feature, at minimum `Ctrl+M` (mnemonic for "map/colormap") should cycle presets.

3. **No loading/empty/error states in viewport** (`viewport.rs:224-232`): When no volume is loaded, the viewport shows a dark-gray border and nothing else. When in depth mode, it shows `"Depth Mode: Seismic Hidden"` -- this is good, but there is no guidance text like `"Import a SEG-Y file to begin"`. No spinner during SEG-Y scanning (the `ImportState::Scanning` state exists but does not render a progress indicator in the viewport).

4. **Hardcoded default indices** (`viewport.rs:40-42`): `inline_idx: 250`, `crossline_idx: 250`, `time_idx: 250` -- these are arbitrary and may be out of range for small surveys. While the slider clamps at render time, the initial displayed value is wrong until a volume is loaded.

5. **Mouse coordinate display is fake** (`app.rs:788`): `"X: 250.5  Y: 312.8  Z: 1523m "` is a hardcoded string, never updated from actual mouse position. This is worse than no coordinate display -- it actively misleads users into thinking the values are real.

6. **Sketch fault mode has visual feedback** (`viewport.rs:136-167`): Green start point, red end point, yellow sketch line. Good visual feedback during sketching.

7. **No right-click context menu**: Standard seismic interpretation tools offer right-click menus for picking, attributes, and export. Not implemented.

## Top Priority Fixes

1. **Remove or fix hardcoded status bar coordinates** (`app.rs:788`, line ~788): The `"X: 250.5  Y: 312.8  Z: 1523m "` text is fake data. Either wire it to actual mouse position tracking or replace with a meaningful indicator (e.g., loaded volume name, CRS, sample count).

2. **Add colormap selector UI** (`viewport.rs`, after line ~107): Add a `egui::ComboBox` or radio group in the toolbar to let users switch between `Seismic`, `Viridis`, `Magma`, `Gray`, `Rainbow`, and `BlueWhiteRed`. Currently the most significant missing UI feature.

3. **Show slider values for Gain and Clip** (`viewport.rs:101-107`): Change `.show_value(false)` to `.show_value(true)` or add a separate label showing the current numeric value. Users cannot tune these parameters effectively without feedback.

## Recommendations (Non-Blocking)

- **Merge Seismic and BlueWhiteRed**: Remove the duplicate `BlueWhiteRed` variant or document it as an explicit alias. Consider renaming `Seismic` to `BlueWhiteRed` since that is the more descriptive name.
- **Add colormap swatch/legend**: Render a thin colorbar strip alongside the viewport showing the current colormap gradient with amplitude scale labels.
- **Implement proper empty state**: When no volume is loaded, show centered text: `"No seismic data loaded. Import a SEG-Y file to begin."` with an import button.
- **Monospace numeric values**: Use `FontFamily::Monospace` for all displayed numbers (coordinates, gain/clip values, slider labels).
- **Colorblind-safe default**: Consider making `Viridis` the default colormap instead of `Seismic` for accessibility.
- **Replace rainbow with cividis**: If a rainbow-like colormap is needed, Cividis is a colorblind-safe alternative to standard rainbow.
- **Add keyboard shortcut hints**: Show keyboard shortcuts in tooltips (e.g., `"Wiggles (toggle)"` with a tooltip showing there is no shortcut yet, encouraging addition).
- **Progress indicator auto-tracking**: Use `egui::Spinner` widget alongside the `ProgressBar` for a more polished loading state.
- **Theme-aware pick outlines**: Use `theme.text_primary` or inverted color for pick point outlines instead of hardcoded white.
