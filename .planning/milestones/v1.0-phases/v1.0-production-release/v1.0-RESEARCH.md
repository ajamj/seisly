# Phase v1.0: Production Release - Research

**Researched:** 2026-03-31
**Domain:** Release Engineering, UI/UX, Performance, Security
**Confidence:** HIGH

## Summary

This research focuses on the final transition of **Seisly** from a feature-rich prototype to a production-grade seismic interpretation platform. The primary goals are ensuring UI/UX consistency, implementing robust error handling and crash reporting, providing professional installers, optimizing performance for large datasets (out-of-core), and securing the Python plugin system.

**Primary recommendation:** Use `cargo-dist` for multi-platform distribution and implement a process-isolated Python worker model with `bubblewrap`/`Job Objects` for plugin security.

## Architecture Patterns

### Recommended Project Structure
```
src/
├── app/               # Main application logic and UI
├── plugins/           # Plugin manager and discovery
├── docs/              # mdBook source
└── packaging/         # Dist-specific assets (icons, licenses)
```

### Pattern 1: Isolated Python Worker
**What:** Spawning a separate process to host the Python interpreter for plugins.
**When to use:** For all Python-based plugin executions to prevent plugin crashes from taking down the main Seisly app.
**Example:**
```rust
// Use gRPC (tonic) or std::process::Command to launch the worker
let mut child = Command::new("seisly-py-worker")
    .arg("--sandbox")
    .spawn()?;
```

### Anti-Patterns to Avoid
- **In-process Python execution:** Running `PyO3` directly in the main UI thread (causes hangs and unrecoverable crashes).
- **Hardcoded Paths:** Avoid hardcoded paths for documentation or assets; use `include_bytes!` or runtime detection.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Installer generation | Custom scripts | `cargo-dist` | Handles multi-arch, multi-OS, and GitHub releases. |

## Common Pitfalls

### Pitfall 1: Memory-Mapping Fragility
**What goes wrong:** Mapping a file that is later deleted or truncated by another process causes a SIGBUS/Access Violation.
**Prevention:** Use a dedicated lock file or file locking (`fs2` crate) to ensure exclusive access to seismic volumes during interpretation.

### Pitfall 2: Python Version Mismatch
**What goes wrong:** Plugin assumes `3.10` features but host provides `3.8`.
**Prevention:** Explicit version check in plugin manifest and use of virtualenvs for plugin isolation.

## Code Examples

### Custom Panic Hook with UI Dialog
```rust
// Source: Community best practice for egui/eframe
fn setup_panic_hook() {
    let default_panic = std::panic::take_hook();
    std::panic::set_hook(Box::new(move |info| {
        let msg = format!("Critical Error: {}", info);
        rfd::MessageDialog::new()
            .set_title("Seisly Crash")
            .set_description(&msg)
            .set_level(rfd::MessageLevel::Error)
            .show();
        default_panic(info);
    }));
}
```

### Process Isolation on Linux (Bubblewrap)
```rust
// Basic bwrap command structure
let output = Command::new("bwrap")
    .arg("--ro-bind").arg("/usr").arg("/usr")
    .arg("--ro-bind").arg("/lib").arg("/lib")
    .arg("--proc").arg("/proc")
    .arg("--dev").arg("/dev")
    .arg("--unshare-all")
    .arg("--new-session")
    .arg("seisly-py-worker")
    .output()?;
```
