## Objective

**Purpose:** `memmap2` is fast but unsafe when the file disappears or the disk fails. This plan adds a safety wrapper.

## Technical Notes

### Why Bounds Checking Instead of Signal Handlers?

1. **Portability**: Signal handling differs between Windows (SEH) and Unix (SIGBUS)
2. **Performance**: Bounds checking is O(1) and predictable
3. **Safety**: Prevention is better than recovery - we never trigger the exception
4. **Simplicity**: No need for complex signal handler setup and teardown

### Design Decisions

- **Cached Length**: Storing the length separately avoids repeated method calls
- **Option-Based API**: Returns `None` for out-of-bounds instead of panicking
- **Extension Trait**: Array reads via trait keeps core API minimal
- **Arc Wrapper**: Thread-safe sharing without exposing raw pointers

### Future Enhancements

1. **Advisory Locking**: Add file locking to prevent concurrent modification
2. **Prefetching**: Hint to OS about access patterns for better performance
3. **Memory Pressure**: Monitor system memory and release mappings when needed
4. **Async Support**: Consider tokio-compatible async mmap wrapper
